# 统计后续问题

## 1. 当前代码已经补到的范围

在不改数据库表结构的前提下，代码层目前已经补齐了以下统计项：

- 车辆指标
  - 车辆总数
  - 车辆类型分布
  - 车辆运能分布
  - 当前货舱载货分布
  - 车辆位置分布（中心点、平均离散距离、最近邻距离、密集/分散判断）
  - 按车辆数利用率
  - 按载重利用率
  - 车辆等待时间统计
  - 空驶距离统计
  - 空驶时间统计（估算）
- 货物指标
  - 总需求量
  - 已分配量
  - 待处理量
  - 总运输量
  - 在途运输量
  - 已完成运输量
  - 收货方运输效率（估算）
  - 总吨距离
  - 个体吨距离统计
  - 总待处理损耗
  - 工厂维度损耗统计
- 其它指标
  - 空满占比
  - 等运占比
  - 装卸时间统计（估算）
  - 当前快照下的全局总成本值 `cost`

## 2. 当前仍然是“估算值”的地方

这些指标已经有代码实现，但本质上仍然是估算，不是严格业务真值：

- 空驶时间
  - 当前做法：`车辆当前位置 -> 任务起点` 的直线距离 / 当前车速
  - 问题：没有真实路径、没有空驶开始时刻
- 收货方运输效率
  - 当前做法：按收货点聚合 `运输吨位 / 运输时长`
  - 在运输中任务上使用当前运输阶段已耗时
  - 在已完成任务上使用 `起终点直线距离 / 车辆速度` 反推时长
  - 问题：缺少真实签收时间和真实运输轨迹
- 装卸时间
  - 当前做法：按 `60 秒 / 吨` 的经验系数估算
  - 运输中订单记 1 次装货
  - 已完成订单记 1 次装货 + 1 次卸货
  - 问题：没有真实装卸事件日志，也没有不同货种/包装/场站的差异参数

## 3. 后续必须解决的问题

### 3.1 任务阶段时间不完整

现在表里只有通用的 `create_time` 和 `update_time`，不足以严格还原调度流程。

后续至少要补齐这些业务时刻：

- 订单被分配车辆时间
- 车辆到达提货点时间
- 开始装货时间
- 完成装货时间
- 开始运输时间
- 到达收货点时间
- 开始卸货时间
- 完成卸货时间
- 签收完成时间

没有这些时间点，下列统计永远只能近似：

- 真实等待时间分段
- 真实运输时间
- 真实装卸时间
- 收货方吞吐效率
- 各阶段 bottleneck 分析

### 3.2 缺少车辆轨迹/里程日志

现在只有车辆当前位置，没有完整轨迹。

后续需要补的能力：

- 车辆轨迹点表或里程日志表
- 每段路径的开始/结束时刻
- 是否空驶、是否载货的阶段标记

没有这些数据，下列统计无法做准：

- 真实空驶距离
- 真实空驶时间
- 真实重载距离
- 真实空满占比
- 车辆空间热区迁移分析

### 3.3 工厂/收货方实体还不够明确

当前代码里：

- 工厂损耗按 `StartPoiId` 聚合
- 收货方效率按 `EndPoiId` 聚合

这只是一个临时代替方案。

后续需要明确：

- `POI` 是否就等于工厂/仓/门店/收货方
- 一个企业是否会对应多个 POI
- 统计维度到底按 POI、按企业，还是按行政区域

如果业务上不是按 POI 统计，后面需要补实体映射关系。

### 3.4 装卸耗时模型需要标定

现在 `60 秒 / 吨` 只是让代码可跑、统计可出数的占位系数。

后续需要解决：

- 不同货物类型的装卸速度不同
- 不同包装方式的装卸速度不同
- 不同站点设备条件不同
- 同一订单可能存在多次装卸/转运

后续应至少把系数改成可配置，再逐步替换为真实事件记录。

### 3.5 全局 cost 需要沉淀为历史序列

现在的总成本值是“当前快照值”，只能看当前时刻系统状态。

后续建议补：

- cost 定时采样
- cost 历史曲线
- 不同成本项 A/B/C/D/E/F 的时间序列
- 调度动作前后 cost 对比

否则很难回答：

- 调度策略有没有变好
- 哪个时段 cost 异常升高
- 哪一类问题在拉高总成本

## 4. 建议的下一步顺序

建议按下面顺序继续做：

1. 先补任务阶段事件时间
2. 再补车辆轨迹或里程日志
3. 明确工厂/收货方统计口径
4. 把装卸系数配置化
5. 最后做统计结果落库或时序化展示
